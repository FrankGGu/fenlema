## 3. 用户核心流程

### 3.1 新用户注册/登录流程
```
1. 微信授权登录 → 2. 完善个人信息 → 3. 创建/加入关系空间 → 4. 邀请伴侣 → 5. 伴侣接受邀请 → 6. 进入首页
```

### 3.2 日常打卡流程
```
1. 用户进入首页 → 2. 点击「今日打卡」→ 3. 填写心情、沟通、亲密度、冲突情况 → 4. 提交 → 5. 同步更新伴侣状态 → 6. 更新关系温度 → 7. 返回首页显示最新状态
```

### 3.3 风险预警与AI介入流程
```
1. 系统检测到漏打卡/低评分 → 2. 首页显示轻提醒 → 3. 连续2天异常 → 4. 进入「关系预警」状态 → 5. 用户点击「寻求帮助」或自动触发 → 6. 进入AI调解室 → 7. AI分别与双方对话（安抚情绪）→ 8. AI识别矛盾类型 → 9. 生成调解建议 → 10. 生成修复任务 → 11. 用户执行任务 → 12. 系统跟踪完成情况
```

### 3.4 关系修复任务流程
```
1. AI生成或系统推荐任务 → 2. 任务推送给双方 → 3. 用户查看任务详情 → 4. 执行任务（如倾听10分钟）→ 5. 提交完成证据（可选）→ 6. 伴侣确认完成 → 7. 任务标记完成 → 8. 获得关系温度提升
```

### 3.5 心情树洞流程
```
1. 用户进入心情树洞 → 2. 选择情绪标签 → 3. 匿名记录感受 → 4. 提交 → 5. 系统保存记录（仅自己可见）→ 6. 推荐相关疗愈内容 → 7. 用户可查看历史记录
```

## 4. 数据表设计

### 4.1 用户表 (users)
| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | int | 主键 |
| openid | varchar(100) | 微信openid |
| nickname | varchar(50) | 昵称 |
| avatar | varchar(255) | 头像URL |
| gender | tinyint | 性别 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 4.2 关系表 (relationships)
| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | int | 主键 |
| user_a_id | int | 用户A ID |
| user_b_id | int | 用户B ID |
| status | tinyint | 关系状态（1:正常 2:预警 3:断开） |
| start_date | date | 关系开始日期 |
| temperature | int | 关系温度（0-100） |
| continuous_days | int | 连续打卡天数 |
| last_checkin_date | date | 最后打卡日期 |
| created_at | datetime | 创建时间 |

### 4.3 打卡记录表 (checkins)
| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | int | 主键 |
| user_id | int | 用户ID |
| relationship_id | int | 关系ID |
| date | date | 打卡日期 |
| mood | tinyint | 心情（1-5，表情） |
| communication | tinyint | 沟通状态（1-5） |
| intimacy | tinyint | 亲密度（1-5） |
| conflict | tinyint | 冲突情况（1-5，1为无冲突） |
| note | text | 备注 |
| created_at | datetime | 创建时间 |

### 4.4 预警记录表 (warnings)
| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | int | 主键 |
| relationship_id | int | 关系ID |
| warning_type | tinyint | 预警类型（1:漏打卡 2:低评分 3:双方冷淡 4:冲突升级） |
| warning_level | tinyint | 预警等级（1:轻提醒 2:中预警 3:高风险） |
| reason | varchar(255) | 预警原因 |
| handled | boolean | 是否已处理 |
| handled_at | datetime | 处理时间 |
| created_at | datetime | 创建时间 |

### 4.5 AI调解记录表 (mediations)

| ai_dialogue_a | text | 与用户A的对话记录 |
| ai_dialogue_b | text | 与用户B的对话记录 |
| conflict_type | varchar(50) | 矛盾类型 |
| suggestion | text | 调解建议 |

| status | tinyint | 状态（1:进行中 2:已完成 3:已关闭） |
| created_at | datetime | 创建时间 |

## 5. 技术架构说明

### 前端技术栈
- 微信小程序原生框架（WXML、WXSS、JavaScript）
- 使用WeUI组件库增强UI一致性
- 自定义组件封装
- 使用wx.setStorageSync进行本地状态缓存

### 后端技术栈（预留）
- Node.js + Express / Python + Flask
- MySQL数据库
- 微信云开发（可选，简化后端）

### 状态管理
- 使用globalData存储全局状态（用户信息、关系状态）
- 页面间通信通过事件总线或全局数据监听

### 模拟数据
- 初期使用本地JSON数据模拟API响应
- 后期可对接真实后端
